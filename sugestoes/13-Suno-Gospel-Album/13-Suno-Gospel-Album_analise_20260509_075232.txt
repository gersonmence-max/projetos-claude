### 1. SQL Injection (queries concatenadas vs parametrizadas)

**Problema:** Queries concatenadas são vulneráveis a ataques de SQL Injection.

**Solução:** Utilizar queries parametrizadas.

**Exemplo:**

**Problema:**
```sql
query = "SELECT * FROM usuarios WHERE nome = '" + nome + "'"
```

**Solução:**
```sql
query = "SELECT * FROM usuarios WHERE nome = %s"
cursor.execute(query, (nome,))
```

### 2. Ausência de transactions em operações críticas

**Problema:** Operações que envolvem múltiplas queries podem falhar parcialmente, deixando o banco de dados em um estado inconsistente.

**Solução:** Utilizar transactions.

**Exemplo:**
```sql
BEGIN TRANSACTION;
TRY:
    # queries aqui
    COMMIT;
EXCEPT:
    ROLLBACK;
```

### 3. Falta de índices em foreign keys e campos de busca

**Problema:** Queries podem ser lentas devido à falta de índices.

**Solução:** Criar índices em foreign keys e campos de busca.

**Exemplo:**
```sql
CREATE INDEX idx_nome ON usuarios (nome);
CREATE INDEX idx_id_pais ON cidades (id_pais);
```

### 4. Uso de SELECT * desnecessário

**Problema:** Selecionar todos os campos pode ser ineficiente e retornar mais dados do que necessário.

**Solução:** Selecionar apenas os campos necessários.

**Exemplo:**

**Problema:**
```sql
SELECT * FROM usuarios
```

**Solução:**
```sql
SELECT id, nome, email FROM usuarios
```

### 5. Ausência de soft delete em dados críticos

**Problema:** Dados importantes podem ser excluídos permanentemente.

**Solução:** Implementar soft delete, adicionando uma coluna `ativo` ou `excluido` para marcar registros como excluídos.

**Exemplo:**
```sql
UPDATE usuarios SET ativo = FALSE WHERE id = %s
```

### 6. Migrations sem rollback

**Problema:** Mudanças no banco de dados podem ser difíceis de reverter.

**Solução:** Implementar migrations com rollback.

**Exemplo:**
```sql
# migration
CREATE TABLE usuarios (
    id INT PRIMARY KEY,
    nome VARCHAR(255)
);

# rollback
DROP TABLE usuarios;
```
