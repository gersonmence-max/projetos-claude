**Algoritmos de assinatura JWT (HS256 vs RS256)**

* HS256 (HMAC com SHA-256) é um algoritmo simétrico que utiliza uma chave secreta compartilhada entre o emissor e o verificador do token.
* RS256 (RSA com SHA-256) é um algoritmo assimétrico que utiliza um par de chaves (pública e privada) para assinar e verificar o token.

Em geral, RS256 é considerado mais seguro do que HS256, pois não requer que a chave secreta seja compartilhada entre as partes.

**Expiração e rotação de tokens**

* É importante definir um tempo de expiração para os tokens para evitar que eles sejam válidos por um período prolongado.
* A rotação de tokens é importante para evitar que um token comprometido possa ser utilizado por um período prolongado.

**Armazenamento seguro (httpOnly cookies vs localStorage)**

* httpOnly cookies são mais seguras do que localStorage, pois não podem ser acessadas por código JavaScript executado no cliente.
* No entanto, é importante notar que httpOnly cookies podem ser vulneráveis a ataques de Cross-Site Scripting (XSS) se o servidor não estiver configurado corretamente.

**Implementação de refresh tokens**

* Refresh tokens são tokens que podem ser utilizados para obter um novo token de acesso quando o token atual expirar.
* É importante implementar refresh tokens para evitar que os usuários precisem se autenticar novamente após a expiração do token de acesso.

**Rate limiting em endpoints de auth**

* É importante implementar rate limiting em endpoints de autenticação para evitar ataques de força bruta.
* O rate limiting pode ser implementado com base no número de tentativas de autenticação em um período de tempo determinado.

**Proteção contra brute force**

* É importante implementar proteção contra ataques de força bruta, como limitar o número de tentativas de autenticação em um período de tempo determinado.

### 1. SQL Injection (queries concatenadas vs parametrizadas)

**Problema:** Queries concatenadas são vulneráveis a ataques de SQL Injection.

**Solução:** Utilizar queries parametrizadas em vez de concatenadas.

**Exemplo:**
```sql
-- Problema
query = "SELECT * FROM usuarios WHERE nome = '" + nome + "'"

-- Solução
query = "SELECT * FROM usuarios WHERE nome = %s"
cursor.execute(query, (nome,))
```

### 2. Ausência de transactions em operações críticas

**Problema:** Operações críticas, como atualizações de dados, podem falhar parcialmente sem transactions.

**Solução:** Utilizar transactions para garantir a integridade dos dados.

**Exemplo:**
```sql
-- Problema
cursor.execute("UPDATE usuarios SET nome = 'Novo Nome' WHERE id = 1")
cursor.execute("UPDATE enderecos SET rua = 'Nova Rua' WHERE id_usuario = 1")

-- Solução
with cursor.connection.cursor() as cursor:
    cursor.execute("BEGIN TRANSACTION")
    try:
        cursor.execute("UPDATE usuarios SET nome = 'Novo Nome' WHERE id = 1")
        cursor.execute("UPDATE enderecos SET rua = 'Nova Rua' WHERE id_usuario = 1")
        cursor.execute("COMMIT")
    except:
        cursor.execute("ROLLBACK")
```

### 3. Falta de índices em foreign keys e campos de busca

**Problema:** Consultas podem ser lentas devido à falta de índices.

**Solução:** Criar índices em foreign keys e campos de busca.

**Exemplo:**
```sql
-- Problema
CREATE TABLE usuarios (
    id INT PRIMARY KEY,
    nome VARCHAR(255)
)

-- Solução
CREATE TABLE usuarios (
    id INT PRIMARY KEY,
    nome VARCHAR(255),
    INDEX (nome)
)
```

### 4. Uso de SELECT * desnecessário

**Problema:** Consultas podem retornar mais dados do que necessário.

**Solução:** Utilizar SELECT com colunas específicas.

**Exemplo:**
```sql
-- Problema
SELECT * FROM usuarios

-- Solução
SELECT id, nome FROM usuarios
```

### 5. Ausência de soft delete em dados críticos

**Problema:** Dados críticos podem ser excluídos permanentemente.

**Solução:** Utilizar soft delete com uma coluna de "excluído" ou "ativo".
